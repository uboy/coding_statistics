# Feature Design Specification: Jira Comprehensive Comments-Period Sheet

## 1) Summary
Проблема: нужен сводный лист в `jira_comprehensive`, где видны все задачи,
по которым были изменения (комментарии) за указанный период, с двумя колонками
комментариев — все комментарии задачи и комментарии в период.
Цель: добавить новый лист в существующий Excel-отчёт `jira_comprehensive` без
создания нового отчёта.
Не цели: изменение логики остальных листов, новые зависимости.

## 2) Scope Boundaries
In scope:
- Новый Excel-лист (например, `Comments_Period`) в `jira_comprehensive`.
- Список задач, у которых были комментарии (созданы или обновлены) в период.
- Две колонки: все комментарии задачи и комментарии в период.
- Поля: issue key, summary, type, status, priority, assignee, created, epic name,
  parent, description, comments, AI_comments, comments in period.

Out of scope:
- Новый отдельный отчёт.
- Изменение формата существующих листов, кроме добавления нового.

## 3) Assumptions + Constraints
- Источник данных: Jira API через `stats_core/reports/jira_comprehensive.py`.
- Комментарии доступны в `issue.fields.comment.comments` с `created` и `updated`.
- Без новых зависимостей.
- Для листа `Comments_Period` требуется выборка по `updated` (или переданный `jql`)
  чтобы включить все задачи с изменениями в период.

## 4) Architecture
Компоненты:
- `stats_core/reports/jira_comprehensive.py`: генерация нового листа.
- Переиспользование уже полученных `all_issues` и комментариев, чтобы не делать
  дополнительных запросов.

Data flow:
Base JQL (existing) -> issues (existing sheets)
Comments_Period JQL (updated-date or provided jql) -> issues -> extract comments
-> filter comments in period -> build sheet.

## 5) Interfaces / Contracts
CLI остаётся прежним:
- `--report jira_comprehensive`
- `--start`, `--end` или `--params jql=...`

Новый лист `Comments_Period` со столбцами:
- `Issue_Key`
- `Summary`
- `Type`
- `Status`
- `Priority`
- `Assignee`
- `Created`
- `Epic_Name`
- `Parent`
- `Description`
- `Comments` (все комментарии задачи)
- `AI_Comments` (обязательно: ИИ-обработка комментариев в период)
- `Comments_In_Period` (только комментарии в период)

Фильтр включения задач:
- Issue включается, если есть хотя бы один комментарий, где
  `created` или `updated` попадает в период.

Период:
- Используются `--start` и `--end` (или период из `params`).
- Границы включительно.
Если даты не заданы, лист `Comments_Period` строится по всем задачам из JQL и
в `Comments_In_Period` включаются все комментарии этих задач.

### AI_Comments generation
Использовать существующую AI-инфраструктуру проекта (как в других Jira-отчётах),
не добавляя зависимостей. Обработка применяется ТОЛЬКО к комментариям, попавшим
в период (`created` или `updated` в пределах периода).

AI по умолчанию ВЫКЛЮЧЕН. Включается явно через параметр:
- `ai_comments_enabled=true` (CLI `--params ai_comments_enabled=true`)

Требования к результату:
- Удалить ссылки, URL, упоминания репозиториев, ссылки на PR/MR, ключи Jira.
- Удалить разметку/форматирование (Markdown/HTML), оставить чистый текст.
- Сформулировать понятный прогресс по задаче:
  - Что сделано.
  - Что планируется.
  - Какие были/есть проблемы или риски.
  - Какие зависимости упомянуты.
- Если данных недостаточно, явно указать нехватку данных вместо выдумывания.

### Prompt requirements
Промпт должен быть максимально структурированным и детальным, чтобы слабые
модели могли дать качественную сводку. Включить:
- Язык ответа: русский.
- Формат JSON с фиксированными полями:
  - `done`, `planned`, `risks`, `dependencies`, `notes`.
- Жёсткие правила: не выдумывать факты, использовать только комментарии периода.
- Ясные правила извлечения: факты "сделано/планируется/риски/зависимости" брать
  только из комментариев; если фрагмент непонятен — отправлять в `notes`.
- Примеры валидного вывода.

## 6) Data Model Changes
Нет.

## 7) Edge Cases + Failure Modes
- Комментарий создан ранее, но обновлён в период -> должен попасть в `Comments_In_Period`.
- В задачах без комментариев лист не включает строку.
- Большие объёмы комментариев -> возможна длинная ячейка; нужно чистить/компактить.
- Если `start/end` не заданы, лист `Comments_Period` включает все комментарии
  задач, попавших в JQL.

## 8) Security Requirements
- Не логировать тела комментариев в stdout, только агрегаты.
- Не включать секреты в отчёт.

## 9) Performance Requirements
- Не делать дополнительных API вызовов.
- Использовать уже загруженные комментарии.

## 10) Observability
- Логировать количество issues в листе и количество комментариев в период.

## 11) Test Plan
- Добавить unit-тест на фильтрацию комментариев по `created/updated`.
- Проверить, что `Comments_In_Period` содержит только комментарии периода.
- Команда: `pytest tests`.

## 12) Rollout + Rollback
Rollout:
- Добавить лист и его сборку в `jira_comprehensive`.

Rollback:
- Удалить новую сборку листа и его регистрацию.

## 13) Acceptance Criteria
- Лист `Comments_Period` появляется в Excel-отчёте.
- В листе присутствуют указанные колонки.
- В `Comments` — все комментарии задачи.
- В `Comments_In_Period` — только комментарии, созданные или обновлённые в период.
- В `AI_Comments` — структурированная сводка по комментариям в период,
  с удалением ссылок/разметки и выделением прогресса, планов, проблем/рисков,
  зависимостей.
- Включены только задачи с комментариями в период.
